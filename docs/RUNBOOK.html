1. 사전 점검

운영 서버에 다음이 설치되어 있어야 합니다.

구성요소버전비고
OpenJDK / Temurin17.0.13 이상21 이상은 미테스트
MariaDB10.6 이상UTF-8, patientinfo DB 생성 필요
Node.js20.x빌드 호스트에만 — 운영 서버는 빌드된 jar만 있으면 됨
pnpm9.15.9 (corepack)전역 설치 금지
클리닉 LAN 내부 trust 모델 가정입니다. 인증·인가 계층이 없으므로 외부에서 접근 가능한 네트워크에 배포하면 안 됩니다.

2. 신규 배포

STEP 1 · 빌드 (CI 또는 빌드 호스트)

git clone <repo> patient_info && cd patient_info
JAVA_HOME=/path/to/jdk-17 PATH=$JAVA_HOME/bin:$PATH ./gradlew build
# 산출물: build/libs/urp-*.jar (프론트엔드 정적 자원 포함)

STEP 2 · 자격증명 환경변수 설정 (운영 호스트)

# 시스템 환경 또는 systemd Service [Service] Environment= 라인
DB_URL="jdbc:mariadb://<db-host>:3306/patientinfo?useSSL=true&serverTimezone=Asia/Seoul"
DB_USERNAME="<rotated_user>"
DB_PASSWORD="<rotated_secret>"
CORS_ALLOWED_ORIGINS="https://<web-host>"   # SPA가 별도 호스트인 경우. same-origin이면 빈 값 OK
fallback이 없습니다. 위 셋 중 하나라도 비어 있으면 Spring이 부팅에 실패합니다. 이건 정상 동작이며, Phase 7 T30이 의도한 fail-fast 보호입니다.

STEP 3 · 실행

java -jar build/libs/urp-0.0.1-SNAPSHOT.jar
# 또는 systemd:  ExecStart=/usr/bin/java -jar /opt/patientinfo/urp.jar

STEP 4 · 헬스체크

curl -fsSL http://<host>:8080/api/v1/patients | head -c 200
curl -i http://<host>:8080/api/v1/patients/__noop__/measurements
# 두 번째 응답은 404, body는 {"status":404,"error":"Not Found","timestamp":"..."}
# 응답 본문에 어떤 patientId/ID도 echo되어선 안 됨

3. 자격증명 로테이션 (필수)

레거시 자격증명 (<legacy>/<legacy>) 이 git 히스토리에 평문으로 영구히 박혀 있습니다. Phase 7이 코드 fallback은 제거했지만, 같은 비밀번호를 쓰는 DB 인스턴스가 존재한다면 즉시 로테이션이 필요합니다.
  1. 새 자격증명 발급 — 길이 16자 이상, 알파넘+기호 혼합. 
    mariadb -u root -p
> ALTER USER 'old_user'@'%' IDENTIFIED BY '<new_secret>';
> FLUSH PRIVILEGES;
  2. 운영 호스트의 DB_PASSWORD 환경변수 교체.
  3. 서비스 재시작 (systemctl restart patientinfo 또는 jar 재기동).
  4. 로그에 HikariPool-1 - Start completed 메시지 확인.
  5. 2단계의 헬스체크 재실행으로 인증 동작 확인.
  6. 구 자격증명 제거: DROP USER 'old_user'@'%'; (충분한 안정화 기간 이후).

4. 프로파일 운용

프로파일활성화속성용도
없음 (=prod) 기본값 모든 자격증명 env로부터, 동일 origin CORS 운영 배포
dev SPRING_PROFILES_ACTIVE=dev 로컬 dev용 fallback, Vite 5173 origin 허용 로컬 개발 (gitignored application-dev.properties 필요)
test JUnit @ActiveProfiles("test") H2 in-memory, 빈 CORS 모든 contract 테스트

dev 프로파일 첫 세팅

cp src/main/resources/application-dev.properties.example \
   src/main/resources/application-dev.properties
# 그리고 REPLACE_WITH_LOCAL_DB_USER / _PASSWORD 두 줄을 본인 로컬 MariaDB에 맞게 수정
# 이 파일은 .gitignore에 등록되어 있어 절대 커밋되지 않습니다.

5. 스모크 테스트

  1. curl http://host:8080/ — SPA가 index.html을 반환해야 함.
  2. curl http://host:8080/api/v1/patients — JSON 배열 반환.
  3. 브라우저에서 /settings 진입 → 테마 토글 → 새로고침 후 유지되는지.
  4. /patients → 환자 클릭 → 세션 클릭 → 그래프와 통계 카드 표시.
  5. 같은 페이지에서 "CSV 내려받기" 동작 확인.
  6. "마스킹 켜짐" 설정 후 환자 목록에서 이름/ID가 마스킹되는지.
  7. 인쇄 미리보기 (Ctrl+P) → chrome (헤더·푸터·버튼) 이 숨겨졌는지 + 브라우저 인쇄 옵션의 "머리글/바닥글" OFF 확인 (URL 노출 방지).

6. 트러블슈팅

증상원인조치
부팅 시 Could not resolve placeholder 'DB_USERNAME' 운영 프로파일이 fallback 없는데 env 미설정 2단계의 세 env 변수 모두 export 후 재기동
모든 API가 500 응답 DB 연결 실패 — Hikari 로그 확인 DB host/port/credential 점검, 방화벽 / firewalld 점검
SPA가 정적 자원 404 빌드된 자원이 jar에 없음 ./gradlew bootJar 재실행 (-PskipFrontend 없이)
SPA가 API 호출 시 CORS 에러 운영에서 SPA가 별도 origin인데 CORS_ALLOWED_ORIGINS 미설정 SPA host를 CSV로 추가 후 서비스 재기동
인쇄 페이지에 환자 ID가 URL footer로 노출 브라우저 인쇄 옵션의 "머리글/바닥글" 이 켜져 있음 인쇄 다이얼로그에서 직접 OFF (CSS로는 차단 불가)
세션 비교가 4개 초과 시 동작 안 함 의도된 cap (MAX_COMPARE_IDS=4) 가독성을 위한 정상 제약 — 변경 시 IMPL_SPEC §8.9 확인

7. 백업 / 복구

DB 단일 노드 백업 — cron 권장:

0 3 * * * mariadb-dump --single-transaction patientinfo \
  | gzip > /var/backups/patientinfo/$(date +\%F).sql.gz

복구:

gunzip -c /var/backups/patientinfo/2026-05-17.sql.gz | mariadb patientinfo
환자 PII가 포함되므로 백업 파일도 동일한 접근 통제·암호화를 적용해야 합니다.

8. 롤백

jar 기반 배포이므로 롤백은 단순:

  1. 이전 jar로 심볼릭 링크 / systemd ExecStart 변경.
  2. systemctl restart patientinfo.
  3. 헬스체크 재실행.

스키마 마이그레이션 (ddl-auto=update)이 비파괴적으로 동작하므로 컬럼 추가 정도는 롤백 안전. 컬럼 삭제·rename 같은 파괴적 변경은 별도 마이그레이션 스크립트 + 백업 후 시도.
