3장. 코드 한 바퀴 둘러보기

실제 파일을 열어 한 계층씩 따라갑니다. Java/Spring을 처음 보더라도 "이게 무슨 의미인지"를 한국어로 옆에 적어두었어요.

3.1 진입점 — UrpApplication.java

Java 프로그램은 무조건 main 메서드가 있는 클래스에서 시작합니다. 이 프로젝트는 src/main/java/com/project/urp/UrpApplication.java 입니다. 

@SpringBootApplication
public class UrpApplication {
    public static void main(String[] args) {
        SpringApplication.run(UrpApplication.class, args);
    }
}
  • @SpringBootApplication — "이 클래스가 있는 폴더와 그 하위 폴더의 모든 컴포넌트(@Controller, @Service, @Repository, @Entity 등)를 자동으로 찾아 등록해라" 라는 메타 표시.
  • SpringApplication.run(...) — Tomcat 웹 서버를 띄우고, DB 연결을 만들고, 모든 컴포넌트의 의존성을 주입한 뒤 요청을 받기 시작.

3.2 Controller 계층 — HTTP의 입구

src/main/java/com/project/urp/controller/MeasurementController.java 

@RestController                       // ① 이 클래스는 HTTP API임 (JSON 반환)
@RequiredArgsConstructor              // ② 아래 final 필드를 받는 생성자를 Lombok이 자동 생성
@RequestMapping("/api/v1")            // ③ 모든 메서드의 URL 앞에 /api/v1 붙임
public class MeasurementController {

    private final MeasurementService measurementService;  // ④ 의존성 주입 (Spring이 알아서 넣어줌)

    @GetMapping("/patients")          // ⑤ GET /api/v1/patients 가 여기로 옴
    public ResponseEntity<List<Patient>> getAllPatients() {
        return ResponseEntity.ok(measurementService.findAllPatients());
    }

    @PostMapping("/patients")         // ⑥ POST /api/v1/patients
    public ResponseEntity<Patient> createPatient(@RequestBody PatientDto patientDto) {
        Patient createdPatient = measurementService.createPatient(patientDto);
        return ResponseEntity.status(HttpStatus.CREATED).body(createdPatient);
    }
    // ... 측정 시작/종료/데이터저장 등도 비슷한 패턴
}
표시 읽는 법
@ 가 붙은 줄 "애너테이션". Spring에게 주는 메타 지시.
@RestController 이 클래스의 모든 메서드는 HTTP 응답을 JSON으로 보낸다.
@GetMapping("/patients") HTTP GET 요청 URL이 매칭되면 이 메서드를 호출.
@RequestBody 요청 본문(JSON)을 Java 객체로 자동 변환해서 받는다.
ResponseEntity.ok(...) HTTP 200 + 응답 본문.

핵심 Controller는 얇아야 합니다. 받아서 Service에 넘기고, Service가 돌려준 결과를 응답으로 감싸는 게 전부. 비즈니스 로직(예: "환자 나이가 음수면 거절") 을 Controller에 쓰지 마세요.

3.3 Service 계층 — 비즈니스 로직

src/main/java/com/project/urp/service/MeasurementService.java 

@Service                          // ① "이 클래스는 비즈니스 로직 컴포넌트" → Spring이 인스턴스 만들어줌
@RequiredArgsConstructor          // ② Repository 3개를 받는 생성자 자동 생성
@Transactional                    // ③ 클래스의 모든 메서드를 DB 트랜잭션으로 감쌈
public class MeasurementService {

    private final MeasurementRepository measurementRepository;
    private final PatientRepository patientRepository;
    private final DataPointRepository dataPointRepository;

    @Transactional(readOnly = true)    // ④ 읽기 전용으로 오버라이드 → DB가 최적화
    public List<Patient> findAllPatients() {
        return patientRepository.findAll();   // ⑤ Repository에 위임
    }

    public Patient createPatient(PatientDto patientDto) {
        // ⑥ 중복 검사 (비즈니스 규칙)
        patientRepository.findByPatientId(patientDto.getPatientId()).ifPresent(p -> {
            throw new IllegalArgumentException("Patient ID already exists: " + p.getPatientId());
        });
        // ⑦ Entity 생성해서 저장
        Patient newPatient = new Patient(
                patientDto.getPatientId(),
                patientDto.getName(),
                patientDto.getAge(),
                patientDto.getSex(),
                patientDto.getHeight(),
                patientDto.getWeight()
        );
        return patientRepository.save(newPatient);
    }
}

여기서 새로운 개념들

  • 트랜잭션 — DB 작업 여러 개가 "전부 성공" 아니면 "전부 취소" 되도록 묶는 단위. @Transactional 메서드가 예외를 던지면 그 메서드에서 한 DB 변경이 자동 롤백됩니다.
  • 읽기 전용 트랜잭션 — 데이터를 바꾸지 않는 메서드는 readOnly = true 를 붙이면 DB가 "어차피 안 바꿀 거니까" 라며 성능 최적화를 합니다.
  • Optional — findByPatientId(...) 가 돌려주는 것은 Optional<Patient>. "있을 수도 없을 수도 있다" 는 뜻. ifPresent(...), orElseThrow(...) 같은 메서드로 안전하게 다룹니다.

3.4 Repository 계층 — DB 접근

src/main/java/com/project/urp/repository/PatientRepository.java 

@Repository
public interface PatientRepository extends JpaRepository<Patient, Long> {
    Optional<Patient> findByPatientId(String patientId);
}

이게 전부입니다. 구현이 없습니다.

  • JpaRepository<Patient, Long> 를 상속하기만 하면 save(), findAll(), findById(), delete() 같은 메서드를 Spring이 자동 제공.
  • findByPatientId(String) 처럼 메서드 이름을 규칙대로 지으면 Spring이 메서드 이름을 파싱해서 SELECT * FROM patients WHERE patient_id = ? SQL을 자동 생성. 직접 SQL을 쓰지 않습니다.

규칙 findByXxx, findByXxxAndYyy, findByXxxOrderByYyyDesc, existsByXxx, countByXxx, deleteByXxx 등. Spring Data JPA 공식 문서의 "Query Methods" 를 검색해 보세요.

3.5 Entity (Domain) 계층 — 테이블 매핑

src/main/java/com/project/urp/domain/Patient.java 

@Entity                           // ① "이 클래스는 DB 테이블"
@Getter                           // ② Lombok: getName(), getAge(), ... 자동 생성
@NoArgsConstructor                // ③ Lombok: 기본 생성자 (JPA가 내부적으로 씀)
@Table(name = "patients")         // ④ 테이블 이름
public class Patient {

    @Id                                                  // ⑤ PK
    @GeneratedValue(strategy = GenerationType.IDENTITY)  // ⑥ DB가 자동 증가
    private Long id;

    @Column(nullable = false, unique = true)
    private String patientId;     // ⑦ 외부에서 쓰는 환자 ID ("p001")

    @Column(nullable = false)
    private String name;
    // ... 나이/성별/키/몸무게도 비슷
}

Measurement.java 에는 한 단계 더 있는데, 관계를 표현합니다. 

@ManyToOne(fetch = FetchType.LAZY)   // ① 다 측정은 한 환자에 속함
@JoinColumn(name = "patient_id")     // ② DB의 patient_id 컬럼이 FK
private Patient patient;

@OneToMany(mappedBy = "measurement", cascade = CascadeType.ALL)
private List<DataPoint> dataPoints = new ArrayList<>();   // ③ 한 측정에는 많은 datapoint
  • @ManyToOne — "내가 N, 상대가 1" (측정 N, 환자 1)
  • @OneToMany — "내가 1, 상대가 N" (측정 1, 데이터포인트 N)
  • fetch = LAZY — 측정을 불러올 때 환자 정보까지는 아직 SELECT 하지 말고, 진짜 필요할 때 가져와라.

3.6 DTO 계층 — 입출력 전용 객체

Entity와 비슷해 보이지만 다릅니다. Entity는 DB 테이블이고, DTO는 요청·응답에 쓰는 단순 데이터 운반체입니다. 

@Getter
@NoArgsConstructor
public class PatientDto {
    private String patientId;
    private String name;
    private Integer age;
    private String sex;
    private Float  height;
    private Float weight;
}

왜 분리? Entity에는 id(내부 PK), 관계 필드, JPA 메타가 붙어있어서 그대로 JSON으로 주고받으면 정보가 새거나 무한 루프가 돕니다. DTO는 클라이언트에게 보여줄 필드만 깔끔하게 모아둡니다.

3.7 글로벌 예외 처리

controller/GlobalExceptionHandler.java 는 어떤 컨트롤러에서든 던져진 예외를 한곳에서 받아 "사용자에게 보여줘도 안전한 형태"의 JSON 응답으로 변환합니다. 

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<Map<String, Object>> handleIllegalArgument(IllegalArgumentException ex) {
        // 메시지에 환자 ID 같은 PII가 있을 수 있으므로 절대 그대로 돌려주지 않음
        log.warn("[{}] IllegalArgumentException (message suppressed for PII)", status.value());
        return error(status);
    }
    // 다른 예외 타입들도 비슷
}

중요한 규칙 새 Service 메서드를 만들 때 에러 메시지에 환자 ID·이름 등 PII를 절대 넣지 마세요. 위 핸들러가 메시지를 가리지만, 같은 메시지가 다른 경로(로그·테스트)로 새는 순간 사고가 됩니다.

3.8 설정 파일 — application.properties

spring.datasource.url=${DB_URL}                 # 환경변수에서 DB 주소 읽음
spring.datasource.username=${DB_USERNAME}
spring.datasource.password=${DB_PASSWORD}
spring.jpa.hibernate.ddl-auto=update            # 부팅 시 엔티티 변경분을 DB에 자동 반영
spring.jackson.serialization.write-dates-as-timestamps=false   # 날짜를 ISO-8601 문자열로
server.error.include-message=never              # 에러 응답에 내부 메시지 포함 X

의료 데이터 주의 ddl-auto=update 는 부팅 시 엔티티 변경을 DB에 자동 ALTER 합니다. 컬럼을 지우면 데이터도 같이 날아갈 수 있다는 뜻. 운영에서는 Flyway 같은 마이그레이션 도구로 전환이 필요합니다 (보안 감사 H-3 참조).

🧪 직접 해보기 — 코드와 응답 매칭

curl -X POST http://localhost:8080/api/v1/patients -H "Content-Type: application/json" -d '{"patientId":"p001","name":"이미있음"}' 를 (같은 ID로) 두 번 보내면 409 Conflict가 떨어집니다.

그 응답이 어디서 만들어졌는지 거꾸로 따라가 보세요: MeasurementService.createPatient 가 IllegalArgumentException 을 던짐 → GlobalExceptionHandler.handleIllegalArgument 가 받음 → 그런데 409가 왜 나오지? 힌트: DataIntegrityViolationException 핸들러.
