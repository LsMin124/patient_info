7장. 문제가 생겼을 때

증상으로 검색해서 해결법을 찾는 사전입니다. 같은 에러 메시지를 10번째 보더라도 이 페이지에서 항상 같은 답을 찾을 수 있어야 합니다.

증상별 인덱스

  • "unsupported class file major version" / java 버전 에러
  • "Port 8080 was already in use"
  • DB 연결 실패 (Hikari)
  • Access denied for user / 비밀번호 틀림
  • "Could not resolve placeholder 'DB_URL'"
  • dev 프로필을 안 켰을 때
  • pnpm command not found
  • CORS 에러 (브라우저 콘솔)
  • 테스트 한두 개만 깨짐
  • Fly.io에 배포는 됐는데 500 에러

"unsupported class file major version" / java 버전 에러

원인: JDK 17 이외 버전으로 빌드 시도. 이 프로젝트는 17을 못 박았습니다.

해결: 

java -version    # "openjdk version 17.x" 가 나와야 함

# SDKMAN 사용 중이면
sdk use java 17.0.13-tem

# 또는 환경변수로
export JAVA_HOME=$HOME/.local/jdks/jdk-17.0.13+11
export PATH=$JAVA_HOME/bin:$PATH

"Port 8080 was already in use"

원인: 이미 다른 백엔드 인스턴스(혹은 다른 앱)가 8080을 잡고 있음. 

# 무엇이 잡고 있는지 보기
sudo lsof -i :8080

# 죽이기
kill -9 <PID>

# 또는 다른 포트로 실행
SERVER_PORT=8090 SPRING_PROFILES_ACTIVE=dev ./gradlew bootRun -PskipFrontend=true

DB 연결 실패 (Hikari)

로그 단서: HikariPool-1 - Exception during pool initialization, Connection refused.

체크리스트:

  1. MariaDB가 켜져 있는가? sudo systemctl status mariadb 또는 brew services list
  2. 포트 3306이 열려 있는가? nc -zv localhost 3306
  3. application-dev.properties 의 URL이 맞는가? (`localhost:3306/patientinfo`)
  4. 데이터베이스 patientinfo 가 실제로 생성되어 있는가? mariadb -u root -e "SHOW DATABASES" | grep patientinfo

Access denied for user / 비밀번호 틀림

로그 단서: Access denied for user 'patient_dev'@'localhost'

해결: application-dev.properties 의 username/password 와 MariaDB에 실제 만들어진 계정/비밀번호가 일치하는지 비교. 비밀번호를 잊었으면 root로 들어가 재설정: 

sudo mariadb -u root
ALTER USER 'patient_dev'@'localhost' IDENTIFIED BY '새비밀번호';
FLUSH PRIVILEGES;

"Could not resolve placeholder 'DB_URL'"

원인: dev 프로필을 켜지 않았는데 DB_URL 환경변수도 없음. 운영 모드는 fallback 없이 fail-fast 하도록 설계됨.

해결: 로컬에서는 dev 프로필을 켭니다. 

SPRING_PROFILES_ACTIVE=dev ./gradlew bootRun -PskipFrontend=true

dev 프로필을 안 켰을 때

증상: 위 DB_URL 에러가 나거나, CORS가 막힘.

해결: 항상 SPRING_PROFILES_ACTIVE=dev 를 앞에 붙이세요. dev 프로필은 application-dev.properties 를 추가로 읽어 로컬 DB 기본값과 Vite CORS origin을 채워줍니다.

pnpm command not found

corepack enable
corepack prepare pnpm@9.15.9 --activate

그래도 안 되면 Node.js 자체가 깔려 있는지 (node -v) 확인.

CORS 에러 (브라우저 콘솔)

증상: Access to fetch at 'http://localhost:8080/api/...' from origin 'http://localhost:5173' has been blocked by CORS

원인 + 해결: dev 프로필이 꺼져 있을 때. dev 프로필이 http://localhost:5173 를 CORS origin에 추가해 줍니다. SPRING_PROFILES_ACTIVE=dev 로 재시작.

테스트 한두 개만 깨짐

  1. build/reports/tests/test/index.html 열기 → 어느 테스트, 어느 줄.
  2. 실패한 케이스의 expected vs actual 비교.
  3. 내가 만진 코드와 그 케이스가 관련이 있는가? Yes → 내 변경이 버그. No → 다른 사람의 변경과 머지 충돌일 수 있음.
  4. 마지막 수단: git stash && ./gradlew test -PskipFrontend=true 로 깨끗한 상태에서 통과 확인 → 내 변경 다시 복원.

Fly.io에 배포는 됐는데 500 에러

fly logs --app patient-info-demo 로 실시간 로그 보기. 자주 만나는 케이스:

  • DB attach 안 됨: postgres attach 가 안 끝났거나 DATABASE_URL 시크릿이 비었음. 위 6.5의 attach 명령 재실행.
  • OOM: 메모리 1GB가 부족. fly scale memory 2048 로 임시 상향. 근본은 메모리 튜닝이 필요할 수 있음.
  • Hibernate ddl-auto 실패: 엔티티와 기존 DB 스키마 충돌. 데모 DB는 fly postgres ssh 후 DROP DATABASE 로 초기화 가능 (실 데이터 X 전제).

일반 디버깅 절차

  1. 전체 에러 메시지를 그대로 복사해 검색. 한 문장만 잘라내면 답을 못 찾습니다.
  2. 직전에 무엇을 바꿨는지 확인 (git diff). 가장 유력한 용의자.
  3. 로컬에서 재현되는가? CI/Fly에서만 나는가? 차이 = 환경변수, JDK 버전, DB 종류.
  4. 그래도 막히면 팀에 질문 — (1)무엇을 했고, (2)무엇이 일어났고, (3)에러 전문, (4)이미 해본 것.
