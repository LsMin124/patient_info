6장. 빌드와 배포

로컬에서 잘 도는 코드를 어떻게 다른 사람의 컴퓨터, 병원 서버, Fly.io 데모에 올리는지 차근차근.

6.1 빌드의 결과물은 무엇인가

./gradlew build 명령은 다음 두 가지 일을 동시에 합니다:

  1. 프런트엔드 빌드 — frontend/ 의 React를 Vite로 빌드해 frontend/dist/ 에 정적 HTML/JS 생성 → src/main/resources/static/ 로 동기화.
  2. 백엔드 빌드 — 자바 코드를 컴파일하고, 위 정적 자원을 포함한 실행 가능한 jar 파일을 build/libs/urp-0.0.1-SNAPSHOT.jar 에 생성.

결과 jar 한 파일이면 백엔드 + 프런트엔드를 동시에 서비스 할 수 있습니다. 별도 웹서버(nginx) 가 필요 없습니다.

6.2 풀빌드 명령

./gradlew build

처음에는 5~10분, 캐시 이후로는 30초 ~ 2분. 마지막에 BUILD SUCCESSFUL 이 보이면 끝.

6.3 jar 직접 실행해 보기

DB_URL='jdbc:mariadb://localhost:3306/patientinfo' \
DB_USERNAME='patient_dev' \
DB_PASSWORD='여기에-비밀번호' \
java -jar build/libs/urp-0.0.1-SNAPSHOT.jar

SPRING_PROFILES_ACTIVE=dev 를 안 붙였으니 운영 프로필로 실행됩니다. DB 환경변수가 없으면 부팅이 일부러 실패합니다(=fallback 제거됨, 안전한 동작).

6.4 Docker 이미지 만들기

docker build -t patient-info:local .

Dockerfile 은 3단계로 동작합니다:

  1. frontend 단계 — node:20-alpine 위에서 pnpm install → pnpm build → frontend/dist 생성.
  2. backend 단계 — Eclipse Temurin 17 위에서 gradle bootJar 실행, 위 dist 미리 stage 해 두기.
  3. runtime 단계 — JRE 17 슬림 이미지에 jar 하나만 복사. 사이즈가 작아짐.

실행 (로컬 컨테이너)

docker run --rm -p 8080:8080 \
  -e DB_URL='jdbc:mariadb://host.docker.internal:3306/patientinfo' \
  -e DB_USERNAME='patient_dev' \
  -e DB_PASSWORD='여기에-비밀번호' \
  patient-info:local

6.5 Fly.io 데모에 배포

전제 Fly.io 배포본은 인터넷에 그대로 노출되며 인증이 없습니다. 절대 실 환자 데이터를 넣지 마세요. 데모/테스트용으로만 쓰세요. (자세한 보안 한계는 SECURITY_AUDIT.md C-1, C-2 참조.)

최초 1회 설정 (이미 되어 있을 가능성 큼)

  1. brew install flyctl (또는 공식 사이트에서 설치)
  2. fly auth login
  3. fly apps create patient-info-demo
  4. Postgres 부착: fly postgres create → fly postgres attach patient-info-demo-db --app patient-info-demo
  5. CORS 시크릿: fly secrets set CORS_ALLOWED_ORIGINS=https://patient-info-demo.fly.dev --app patient-info-demo
  6. GitHub Actions 자동 배포를 쓰려면 FLY_API_TOKEN 을 레포 시크릿에 추가 ( fly tokens create deploy 로 생성).

한 번에 배포

fly deploy --remote-only

또는 main 브랜치에 push 하면 GitHub Actions가 자동 배포.

상태 확인 명령

명령의미
fly status --app patient-info-demo 머신 상태
fly logs --app patient-info-demo 실시간 로그
fly ssh console --app patient-info-demo 컨테이너 안으로 접속
fly scale count 0 --app patient-info-demo 완전 OFF ($0)
fly scale count 1 --app patient-info-demo 다시 ON

6.6 CI/CD 흐름 한눈에

.github/workflows/ci.yml 이 다음 일을 자동화합니다:

  • push 또는 PR 발생 시 트리거
  • JDK 17, Node 20, pnpm 9.15.9 셋업
  • 프런트: tsc / lint / vitest / vite build
  • 백엔드: ./gradlew build (테스트 22/22 + jar)
  • main 브랜치 push 시 추가로 flyctl deploy --remote-only 실행 (FLY_API_TOKEN 필요)

팁 PR을 만들면 CI 결과 배지가 PR 페이지에 붙습니다. 빨강이면 머지 금지. 어디서 깨졌는지 확인 후 7장 트러블슈팅 참고.

6.7 GitHub Pages 로 이 매뉴얼 공개하기

지금 보고 있는 매뉴얼은 docs/manual/ 하위에 정적 HTML로 작성되어 GitHub Pages 로 그대로 서비스됩니다.

  1. 레포 Settings → Pages
  2. Source: Deploy from a branch
  3. Branch: main, Folder: /docs
  4. Save → 1~2분 뒤 https://<owner>.github.io/patient_info/manual/ 로 접속.

docs/manual/*.html 을 수정해 push 하면 자동으로 갱신됩니다 (Actions 워크플로 없이 동작).

🧪 직접 해보기 — 풀빌드

./gradlew clean build

마지막 BUILD SUCCESSFUL이 보이면 OK. 그 다음 build/libs/ 에 jar 가 생긴 걸 확인하세요. 그 jar이 운영에 올라가는 결과물입니다.
