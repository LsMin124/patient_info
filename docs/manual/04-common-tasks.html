4장. 자주 하는 작업

가장 많이 받게 될 요청 4가지를 단계별로. 각 작업은 1시간 안에 끝나는 크기로 잘랐습니다.

4.1 새 API 엔드포인트 추가하기

예시: "환자 한 명의 상세정보를 GET 으로 조회하는 엔드포인트가 필요해" 라는 요청.

1단계 — Controller에 메서드 추가

src/main/java/com/project/urp/controller/MeasurementController.java 

@GetMapping("/patients/{patientId}")
public ResponseEntity<Patient> getPatientByPatientId(@PathVariable String patientId) {
    Patient patient = measurementService.findPatientByPatientId(patientId);
    return ResponseEntity.ok(patient);
}

2단계 — Service에 메서드 추가

src/main/java/com/project/urp/service/MeasurementService.java 

@Transactional(readOnly = true)
public Patient findPatientByPatientId(String patientId) {
    return patientRepository.findByPatientId(patientId)
        .orElseThrow(() -> new IllegalArgumentException("Invalid patient Id: " + patientId));
}

왜 "Invalid patient Id" 라는 정확한 문구? GlobalExceptionHandler 가 이 prefix를 보고 404로 매핑합니다 (3.7 참고). 새 메시지를 만들 때는 기존 패턴을 따라가세요. (장기적으로는 도메인 예외 클래스로 바꾸는 게 안전 — SECURITY_AUDIT M-2 참조.)

3단계 — Repository는 이미 있음

findByPatientId(String) 가 이미 PatientRepository 에 정의되어 있어 추가 작업 없음.

4단계 — 테스트 추가 (5장 참고)

src/test/java/com/project/urp/controller/PatientApiContractTest.java 에 한두 케이스를 추가하세요. 자세한 작성법은 5장.

5단계 — 동작 확인

# 백엔드 재시작
SPRING_PROFILES_ACTIVE=dev ./gradlew bootRun -PskipFrontend=true

# 다른 터미널에서
curl http://localhost:8080/api/v1/patients/p001
curl -i http://localhost:8080/api/v1/patients/존재하지않는ID  # 404 떨어지는지

4.2 환자 테이블에 새 컬럼 추가하기

예시: "환자 비고(note) 필드를 저장하고 싶다"

1단계 — Entity에 필드 추가

src/main/java/com/project/urp/domain/Patient.java 

@Column(length = 500)   // nullable 기본값 = true. 필수면 nullable = false 추가

생성자에도 추가하세요. (Lombok @NoArgsConstructor 만 있고 직접 만든 생성자가 있으니, 그쪽 시그니처에 새 파라미터를 넣어주세요.)

@Getter
@NoArgsConstructor
public class PatientDto {
    // 기존 필드들...
    private String note;    // ← 추가

3단계 — Service의 createPatient 에서 전달

Patient newPatient = new Patient(

    patientDto.getAge(),
    patientDto.getSex(),

4단계 — DB 컬럼 생성

spring.jpa.hibernate.ddl-auto=update 가 켜져 있어 백엔드 재시작만 하면 자동으로 ALTER TABLE patients ADD note VARCHAR(500) 이 실행됩니다.

운영 DB에서는 주의 ddl-auto=update 는 컬럼 추가는 안전하지만 컬럼 삭제·타입 변경 시 데이터를 잃을 수 있습니다. 운영 DB는 백업 → 점검 시간 → 마이그레이션 순서로 진행하고, 모르겠으면 팀 리드에게 물어보세요.

4.3 입력값 검증 추가하기 (예: 나이는 0~150)

현재 코드에는 검증이 거의 없어 음수 나이나 빈 이름이 통과합니다 (SECURITY_AUDIT H-1). 안전하게 막는 법:

방법 A — Service에서 직접 검증 (당장 가능)

public Patient createPatient(PatientDto patientDto) {
    // 검증
    if (patientDto.getPatientId() == null || patientDto.getPatientId().isBlank()) {
        throw new IllegalArgumentException("patientId must not be empty");
    }
    if (patientDto.getAge() == null || patientDto.getAge() < 0 || patientDto.getAge() > 150) {
        throw new IllegalArgumentException("age must be between 0 and 150");
    }
    // ... 나머지 기존 로직
}

방법 B — Bean Validation 도입 (권장, 추후 작업)

build.gradle 에 implementation 'org.springframework.boot:spring-boot-starter-validation' 추가 후 DTO에 @Min/@Max/@NotBlank 붙이고 Controller에 @Valid. SECURITY_AUDIT H-1 의 권장 조치 항목이기도 합니다.

4.4 응답에 새 필드 더하기 (Measurement에 dataPoint 개수)

예시: 측정 세션 목록에 각 세션의 데이터포인트 개수를 함께 보여주고 싶음.

1단계 — DTO에 필드 추가

src/main/java/com/project/urp/dto/MeasurementSummaryDto.java 

@Getter
public class MeasurementSummaryDto {
    private Long measurementId;
    private LocalDateTime startTime;
    private LocalDateTime endTime;
    private String memo;
    private Integer dataPointCount;     // ← 추가

    public MeasurementSummaryDto(Measurement measurement) {
        this.measurementId = measurement.getId();
        this.startTime = measurement.getStartTime();
        this.endTime = measurement.getEndTime();
        this.memo = measurement.getMemo();
        this.dataPointCount = measurement.getDataPoints().size();   // ← 추가
    }
}

2단계 — 그게 끝

findMeasurementByPatient 가 이미 이 DTO를 생성자로 래핑하고 있어 추가 작업 없음. 백엔드 재시작 → 호출 → dataPointCount 가 응답에 보이면 끝.

성능 함정 measurement.getDataPoints() 는 LAZY 로딩이라 호출 순간 측정 한 건마다 별도 SELECT 가 나갑니다(N+1 문제). 세션이 많아지면 페이지가 느려지므로, 진짜 필요해지면 DataPointRepository.countByMeasurement_Id(id) 같은 카운트 쿼리로 바꾸세요. 지금은 신경 안 써도 됩니다.

4.5 로그 켜고 끄기

기본은 SQL 로그가 꺼져 있어 환자 데이터가 콘솔에 새지 않습니다. 디버깅이 정말 필요하면 임시로: 

# application-dev.properties 끝에 임시 추가 (절대 운영 파일에는 X)
spring.jpa.show-sql=true
spring.jpa.properties.hibernate.format_sql=true
logging.level.org.hibernate.SQL=DEBUG

4.6 작업이 끝났을 때 (커밋 전 체크리스트)

  • ./gradlew test -PskipFrontend=true — 모든 백엔드 테스트 통과?
  • ./gradlew build — 풀빌드 통과? (프런트 포함)
  • 새로 추가한 메서드에 테스트 1개 이상?
  • 임시로 켰던 SQL 로그·디버그 옵션 되돌렸나?
  • 비밀번호·환자 PII가 실수로 커밋되지 않았나? git diff --staged 로 확인.
