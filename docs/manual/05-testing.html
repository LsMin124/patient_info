5장. 테스트 다루기

테스트는 "내가 만든 변경이 다른 곳을 깨뜨리지 않았다" 는 안전망입니다. 이 장은 기존 테스트 실행 + 새 테스트 추가까지를 다룹니다.

5.1 테스트가 사는 곳

src/test/java/com/project/urp/
├── UrpApplicationTests.java                      ← 부팅 smoke test
├── config/
│   └── DatabaseUrlEnvironmentPostProcessorTest.java
├── controller/
│   ├── CorsConfigTest.java
│   ├── GlobalExceptionHandlerTest.java
│   ├── PatientApiContractTest.java               ← 환자 API
│   ├── MeasurementLifecycleContractTest.java     ← 측정 라이프사이클
│   └── ReadApiContractTest.java                  ← 읽기 API
└── support/
    └── ApiContractTestBase.java                  ← 공용 베이스 클래스

5.2 테스트 한 번에 다 돌리기

./gradlew test -PskipFrontend=true

-PskipFrontend=true 를 빼면 프런트엔드까지 빌드되어 훨씬 느려집니다. 백엔드 테스트만 보려면 항상 이 옵션을 쓰세요.

마지막에 다음과 비슷한 줄이 보이면 통과:

BUILD SUCCESSFUL in 24s
22 tests completed, 0 failed

5.3 특정 테스트 클래스 하나만

./gradlew test --tests "com.project.urp.controller.PatientApiContractTest"

5.4 테스트가 깨졌을 때 보는 곳

  • 터미널의 빨간 줄 (PatientApiContractTest > createPatient_duplicate FAILED) 이 어디서 실패했는지 알려줍니다.
  • 자세한 리포트는 build/reports/tests/test/index.html 파일을 브라우저로 여세요. 어느 줄에서 어떤 값이 다른지 깔끔하게 보여줍니다.

5.5 새 테스트 한 개 추가해 보기

4.1 에서 추가한 GET /api/v1/patients/{patientId} 엔드포인트에 테스트를 한 개 붙입시다.

src/test/java/com/project/urp/controller/PatientApiContractTest.java 를 열고 클래스 안에 다음 메서드를 추가: 

@Test
@DisplayName("GET /patients/{patientId} — 존재하면 200 + 환자 정보")
void getPatient_existingId_returnsOk() throws Exception {
    // Arrange: 미리 한 명 등록
    mockMvc.perform(post("/api/v1/patients")
            .contentType(MediaType.APPLICATION_JSON)
            .content("""
                {"patientId":"p001","name":"홍길동","age":30,
                 "sex":"M","height":170,"weight":65}
                """))
        .andExpect(status().isCreated());

    // Act + Assert
    mockMvc.perform(get("/api/v1/patients/p001"))
        .andExpect(status().isOk())
        .andExpect(jsonPath("$.patientId").value("p001"))
        .andExpect(jsonPath("$.name").value("홍길동"));
}

@Test
@DisplayName("GET /patients/{patientId} — 없으면 404")
void getPatient_missingId_returns404() throws Exception {
    mockMvc.perform(get("/api/v1/patients/없는ID"))
        .andExpect(status().isNotFound());
}

이 코드의 핵심 부품

  • @Test — JUnit 5의 테스트 표시.
  • @DisplayName — 한국어로 의도를 적어 두면 리포트가 사람이 읽기 좋게 됩니다.
  • mockMvc.perform(...) — 실제 HTTP 서버를 띄우지 않고 가짜 요청을 보내는 도구. ApiContractTestBase 가 이미 세팅해 둠.
  • jsonPath("$.patientId") — JSON 응답에서 특정 필드를 꺼내 검증.

5.6 테스트 DB는 어디서 오나

src/test/resources/application-test.properties 가 H2 (메모리 DB) 를 사용하도록 설정되어 있어, 테스트 실행 시 로컬 MariaDB와 무관하게 깨끗한 메모리 DB가 만들어졌다 사라집니다. 즉 테스트는 본인 노트북의 환자 데이터를 건드리지 않습니다.

5.7 새 기능을 만들 때 (TDD 권장 흐름)

  1. Red — 먼저 실패하는 테스트를 작성. 아직 구현이 없으니 컴파일 에러 또는 빨강.
  2. Green — Controller/Service/Repository에 가장 단순한 구현을 만들어 테스트가 초록색이 되게 함.
  3. Refactor — 동작은 유지하면서 코드를 깔끔하게 정리. 테스트가 안전망 역할.

팁 처음에는 어색해도, 작은 단위로 한두 번 해보면 "수정하다 뭔가 깨졌을 때 어디부터 봐야 할지" 감각이 빨리 잡힙니다.

5.8 커버리지 보기

현재 프로젝트는 별도 커버리지 도구가 백엔드에 붙어있지 않습니다. 테스트 개수만 통과 기준입니다 (CI 22/22).

프런트엔드는 cd frontend && pnpm test:coverage 로 볼 수 있고 frontend/coverage/index.html 에 결과가 생깁니다.

🧪 직접 해보기 — 일부러 깨뜨리기

./gradlew test -PskipFrontend=true --tests "com.project.urp.controller.PatientApiContractTest"
